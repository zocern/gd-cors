## 1. 功能概述

在原有文件管理基础上，追加**文件版本管理**能力：

- **上传新文件**：自动创建 v1 版本记录。
- **更新文件内容**：不再覆盖旧文件，而是追加新版本（v2、v3…），旧版本 MinIO 文件保留（支持下载历史版本），旧版本 Milvus 向量数据删除（Milvus 始终只保留当前激活版本的向量）。
- **查看版本列表**：可查看某文件的所有历史版本及其向量化状态。
- **切换/回滚版本**：可切换到任意历史版本，系统自动异步完成向量库的新旧版本替换，**保证向量库中有且只有当前激活版本的数据**。
- **删除文件**：彻底删除时，所有历史版本的 MinIO 文件和向量数据一并清理。

## 2. 数据库变更

### 2.1 `file_metadata` 表追加字段

```sql
ALTER TABLE `file_metadata`
    ADD COLUMN `current_version` INT UNSIGNED NOT NULL DEFAULT 1
        COMMENT '当前激活的版本号（从1开始），文件夹为 NULL',
    ADD COLUMN `version_count`   INT UNSIGNED NOT NULL DEFAULT 1
        COMMENT '总版本数（冗余字段，避免 COUNT 查询），文件夹为 NULL';
```

> **存量数据**：`DEFAULT 1` 保证所有已有文件行自动成为 v1，无需手动处理。

### 2.2 新增 `file_versions` 版本表

```sql
CREATE TABLE IF NOT EXISTS `file_versions`
(
    `id`          BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY COMMENT '版本记录ID',
    `file_id`     BIGINT UNSIGNED NOT NULL COMMENT '关联的 file_metadata.id',
    `version`     INT UNSIGNED    NOT NULL COMMENT '版本号（从1开始递增）',
    `storage_key` VARCHAR(255)    NOT NULL COMMENT '该版本在 MinIO 的存储键',
    `size`        BIGINT UNSIGNED NOT NULL DEFAULT 0 COMMENT '该版本文件大小（字节）',
    `md5`         CHAR(32)        NULL COMMENT '该版本文件 MD5',
    `remark`      VARCHAR(500)    NULL COMMENT '版本备注（可选）',
    `created_by`  BIGINT          NULL COMMENT '上传者 ID',
    `created`     DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '版本创建时间',
    UNIQUE KEY `uk_file_version`  (`file_id`, `version`),
    UNIQUE KEY `uk_storage_key`   (`storage_key`),
    INDEX         `idx_file_id`   (`file_id`)
) COMMENT '文件版本表'
  ENGINE = InnoDB
  DEFAULT CHARSET = utf8mb4
  COLLATE = utf8mb4_unicode_ci;
```

### 2.3 `file_vector_status` 表无需变更

`file_vector_status` 以 `storage_key` 为主键，天然支持多版本（每个版本有独立的 `storage_key`，对应独立的状态记录）。

## 3. 新增 / 修改接口

### 3.1 新增接口

所有接口前缀：`/api/v1/files`，需登录（携带 `Authorization` Header）。

#### `GET /files/{id}/versions` — 查询文件版本列表

**请求参数**

| 参数 | 位置 | 类型 | 必填 | 说明 |
|------|------|------|------|------|
| id | Path | Long | 是 | 文件ID（file_metadata.id） |

**响应示例**

```json
{
  "code": 200,
  "data": [
    {
      "id": 1,
      "fileId": 100,
      "version": 1,
      "size": 204800,
      "md5": "d41d8cd98f00b204e9800998ecf8427e",
      "remark": null,
      "createdBy": 1,
      "created": "2026-06-01T10:00:00",
      "current": false,
      "vectorStatus": "SUCCESS"
    },
    {
      "id": 2,
      "fileId": 100,
      "version": 2,
      "size": 307200,
      "md5": "a87ff679a2f3e71d9181a67b7542122c",
      "remark": null,
      "createdBy": 1,
      "created": "2026-06-10T14:30:00",
      "current": false,
      "vectorStatus": "SUCCESS"
    },
    {
      "id": 3,
      "fileId": 100,
      "version": 3,
      "size": 512000,
      "md5": null,
      "remark": null,
      "createdBy": 1,
      "created": "2026-06-19T09:00:00",
      "current": true,
      "vectorStatus": "SUCCESS"
    }
  ]
}
```

**字段说明**

| 字段 | 说明 |
|------|------|
| `current` | 是否为当前激活版本 |
| `vectorStatus` | 该版本的向量化状态：`PENDING` / `PROCESSING` / `SUCCESS` / `FAILED` / `UNSUPPORTED` / `null`（未知） |

---

#### `PUT /files/{id}/versions/switch` — 切换到指定版本

**请求参数**

| 参数 | 位置 | 类型 | 必填 | 说明 |
|------|------|------|------|------|
| id | Path | Long | 是 | 文件ID |

**请求体**

```json
{
  "version": 1
}
```

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| version | Integer | 是 | 目标版本号（≥1） |

**响应示例**

```json
{
  "code": 200,
  "data": null
}
```

**错误码**

| 场景 | HTTP 状态 | code | 说明 |
|------|-----------|------|------|
| 文件不存在 | 200 | 404 | 文件ID无效 |
| 目标版本不存在 | 200 | 404 | 版本号不存在 |
| 已是当前版本 | 200 | 400 | 无需切换 |
| 对文件夹操作 | 200 | 400 | 文件夹不支持版本切换 |

### 3.2 修改接口

#### `POST /files/update` — 更新文件内容（行为变更）

**变更前**：覆盖旧文件，删除旧向量，触发新向量化。
**变更后**：追加新版本，旧版本 MinIO 文件保留（支持下载历史版本），旧版本 Milvus 向量删除，触发新版本向量化。

接口签名、请求参数、响应格式**均不变**，对前端透明。

---

#### `DELETE /files/{id}` — 删除文件（行为变更）

**变更前**：仅删除当前激活版本的 MinIO 文件和向量数据。  
**变更后**：删除**所有历史版本**的 MinIO 文件和向量数据。

接口签名、请求参数、响应格式**均不变**，对前端透明。

---

#### `GET /files/{id}` 和 `GET /files` — 文件信息（响应字段新增）

`FileMetadataVo` 新增两个字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| `currentVersion` | Integer | 当前激活版本号，文件夹为 null |
| `versionCount` | Integer | 总版本数，文件夹为 null |

## 4. 核心流程说明

### 4.1 上传新文件

```
POST /files/upload
    │
    ▼
上传文件到 MinIO（storageKey = UUID + 扩展名）
    │
    ▼
saveFileRecordWithTransaction()  [事务]
    ├── 插入 file_metadata（currentVersion=1, versionCount=1）
    ├── 插入 file_versions（fileId, version=1, storageKey）
    └── 插入 file_vector_status（storageKey, status=PENDING）
    │
    ▼
发送 FileVectorMessage 到 RabbitMQ → 异步向量化
```

### 4.2 更新文件内容（追加新版本）

```
POST /files/update
    │
    ▼
上传新文件到 MinIO（newStorageKey）
    │
    ▼
appendVersionWithTransaction()  [事务]
    ├── 插入 file_versions（version=N+1, storageKey=newStorageKey）
    ├── 更新 file_metadata（storageKey=newStorageKey, currentVersion=N+1, versionCount=N+1）
    ├── 插入 file_vector_status（newStorageKey, status=PENDING）
    └── 注册 afterCommit 钩子
    │
    ▼（事务提交后）
    ├── 发送 FileVectorMessage → 异步向量化新版本
    └── 发送 FileDeleteMessage（deleteStatus=false, storageKey=oldStorageKey）→ 删除旧版本 Milvus 向量
        （旧版本 MinIO 文件保留，支持下载历史版本；Milvus 始终只保留当前激活版本的向量）
```

### 4.3 版本切换/回滚

```
PUT /files/{id}/versions/switch  { "version": N }
    │
    ▼
switchVersion()  [事务]
    ├── 查询目标版本记录（file_versions）
    ├── 更新 file_metadata（storageKey=目标版本storageKey, currentVersion=N）
    └── 注册 afterCommit 钩子
    │
    ▼（事务提交后）
triggerVectorSwitch()  [异步]
    ├── 发送 FileDeleteMessage（deleteStatus=false）→ 删除当前激活版本 Milvus 向量数据
    │   （保留 file_vector_status 记录）
    │
    └── 重置目标版本 file_vector_status 为 PENDING
        发送 FileVectorMessage → 重新向量化目标版本
        （Milvus 只维护当前激活版本，历史版本向量在文件更新时已删除，切换时必须重新向量化）
```

### 4.4 删除文件（清理所有版本）

```
DELETE /files/{id}
    │
    ▼
deleteWithTransaction()  [事务]
    ├── 查询所有后代节点（含子文件夹）
    ├── 查询所有文件节点的全部版本（file_versions）
    ├── 删除 file_versions 记录
    ├── 删除 file_metadata 记录
    ├── 删除 file_association 记录
    └── 注册 afterCommit 钩子
    │
    ▼（事务提交后）
    对每个历史版本的 storageKey：
    ├── 发送 FileDeleteMessage（deleteStatus=true）→ 删除 MinIO 文件
    └── 发送 FileDeleteMessage（deleteStatus=true）→ 删除 Milvus 向量 + 状态表记录
```

### 4.5 向量库始终只有当前版本

> **设计原则**：MinIO 保留所有历史版本文件（支持下载），Milvus 只保留当前激活版本的向量（节省存储，保证 RAG 检索准确）。

| 操作 | 旧版本向量（Milvus） | 新版本向量（Milvus） |
|------|-----------|-----------|
| 上传新文件 | — | 触发向量化 |
| 更新文件（追加版本） | **删除** | 触发向量化 |
| 切换到历史版本 | 删除（当前激活版本的向量） | **重新向量化** |
| 删除文件 | 全部删除 | 全部删除 |

## 6. 存量数据无缝迁移方案

### 6.1 整体原则

- **不停机**：所有变更均为追加操作（新增字段有默认值、新增表），不修改现有数据结构。
- **不丢数据**：存量文件自动成为 v1，MinIO 文件和向量数据无需任何处理。
- **分步执行**：先执行 DDL，再部署新代码，最后执行数据迁移 SQL。

### 6.2 执行步骤

#### Step 1：执行 DDL（可在线执行，不锁表）

```sql
-- 1. file_metadata 追加字段（有默认值，秒级完成）
ALTER TABLE `file_metadata`
    ADD COLUMN `current_version` INT UNSIGNED NOT NULL DEFAULT 1
        COMMENT '当前激活的版本号',
    ADD COLUMN `version_count`   INT UNSIGNED NOT NULL DEFAULT 1
        COMMENT '总版本数';

-- 2. 创建 file_versions 表
CREATE TABLE IF NOT EXISTS `file_versions`
(
    `id`          BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    `file_id`     BIGINT UNSIGNED NOT NULL,
    `version`     INT UNSIGNED    NOT NULL,
    `storage_key` VARCHAR(255)    NOT NULL,
    `size`        BIGINT UNSIGNED NOT NULL DEFAULT 0,
    `md5`         CHAR(32)        NULL,
    `remark`      VARCHAR(500)    NULL,
    `created_by`  BIGINT          NULL,
    `created`     DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP,
    UNIQUE KEY `uk_file_version` (`file_id`, `version`),
    UNIQUE KEY `uk_storage_key`  (`storage_key`),
    INDEX `idx_file_id` (`file_id`)
) ENGINE = InnoDB DEFAULT CHARSET = utf8mb4 COLLATE = utf8mb4_unicode_ci
  COMMENT '文件版本表';
```

#### Step 2：部署新版本代码

此时新代码已运行，新上传/更新的文件会正确写入 `file_versions`。存量文件的 `file_versions` 记录尚未补全，但不影响现有功能（下载、向量化等仍走 `file_metadata.storage_key`）。

#### Step 3：执行存量数据迁移 SQL

将现有文件的 `storage_key` 作为 v1 写入 `file_versions`：

```sql
-- 迁移存量文件数据（仅处理文件，跳过文件夹和 storage_key 为空的记录）
-- 使用 INSERT IGNORE 防止重复执行报错
INSERT IGNORE INTO `file_versions`
    (`file_id`, `version`, `storage_key`, `size`, `md5`, `created_by`, `created`)
SELECT
    `id`,
    1,
    `storage_key`,
    COALESCE(`size`, 0),
    `md5`,
    `created_by`,
    `created`
FROM `file_metadata`
WHERE `folder` = 0
  AND `storage_key` IS NOT NULL
  AND `storage_key` != '';
```

> **说明**：
> - `INSERT IGNORE` 保证迁移 SQL 可重复执行（幂等）。
> - `file_metadata.current_version` 和 `version_count` 已通过 `DEFAULT 1` 自动设置，无需额外更新。
> - 迁移完成后，所有存量文件均有对应的 v1 版本记录，版本管理功能对存量数据完全生效。

#### Step 4：验证

```sql
-- 验证：file_metadata 中的文件数量应等于 file_versions 中 version=1 的记录数
SELECT
    (SELECT COUNT(*) FROM file_metadata WHERE folder = 0 AND storage_key IS NOT NULL AND storage_key != '') AS file_count,
    (SELECT COUNT(*) FROM file_versions WHERE version = 1) AS v1_count;

-- 两个数值应相等
```

### 6.3 MinIO 数据无需处理

存量文件的 MinIO 对象已存在，`storage_key` 不变，新功能直接复用，**无需任何 MinIO 侧操作**。

### 6.4 向量库数据无需处理

存量文件的向量数据已在 Milvus 中，`file_vector_status` 记录已存在，无需任何处理。

> **注意**：存量文件只有 v1 一个版本，不存在版本切换场景，因此不受影响。若后续对存量文件执行"更新文件"操作，新版本会正常走向量化流程，旧版本（v1）的 Milvus 向量会被删除，行为与新文件完全一致。

## 附：版本生命周期示意

```
上传 report.pdf
  file_metadata: storageKey=key-A, currentVersion=1, versionCount=1
  file_versions: (v1, key-A)
  MinIO:         key-A ✓
  Milvus:        key-A ✓

更新文件（上传新内容）
  → 删除 key-A 的 Milvus 向量
  file_metadata: storageKey=key-B, currentVersion=2, versionCount=2
  file_versions: (v1, key-A), (v2, key-B)
  MinIO:         key-A ✓  key-B ✓
  Milvus:        key-B ✓

再次更新
  → 删除 key-B 的 Milvus 向量
  file_metadata: storageKey=key-C, currentVersion=3, versionCount=3
  file_versions: (v1, key-A), (v2, key-B), (v3, key-C)
  MinIO:         key-A ✓  key-B ✓  key-C ✓
  Milvus:        key-C ✓

切换回 v1（回滚）
  → 删除 key-C 的 Milvus 向量
  → 重新向量化 key-A
  file_metadata: storageKey=key-A, currentVersion=1
  MinIO:         key-A ✓  key-B ✓  key-C ✓
  Milvus:        key-A ✓

切换到 v2
  → 删除 key-A 的 Milvus 向量
  → 重新向量化 key-B
  file_metadata: storageKey=key-B, currentVersion=2
  MinIO:         key-A ✓  key-B ✓  key-C ✓
  Milvus:        key-B ✓

删除文件
  → 删除 file_metadata、file_versions、file_association
  → 删除 key-A、key-B、key-C 的 MinIO 文件
  → 删除 key-A、key-B、key-C 的 Milvus 向量 + file_vector_status 记录
```
