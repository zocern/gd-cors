## 3. 第一步：编译后端 JAR 包

> 在**本地开发机**上执行，需要 JDK 17 + Maven 3.9+。

```bash
# 在项目根目录执行
cd /path/to/gd-cors

# 清理并打包，跳过测试
mvn clean package -DskipTests

# 确认编译产物
ls target/gd-cors-*.jar
# 输出示例：target/gd-cors-0.0.1-SNAPSHOT.jar
```

---

## 4. 第二步：准备 Docker 部署目录

编译完成后，将 JAR 整理到 `docker/app-gd-cors/` 目录，服务器上直接用 JAR 构建镜像，**不需要把源码或 pom.xml 带过去**。

> **关于复用项目根目录的 `Dockerfile`**：项目根目录的 `Dockerfile` 是多阶段构建，Run 阶段写的是 `COPY --from=build /app/target/*.jar app.jar`，依赖 Build 阶段（Maven 编译）的产物，无法单独接收外部 JAR。因此需要一个**精简版 Dockerfile**，只保留 Run 阶段，直接 `COPY` 本地编译好的 JAR。

### 4.1 准备后端构建上下文

```bash
cd /path/to/gd-cors

# 创建后端构建上下文目录
mkdir -p docker/app-gd-cors

# 将编译好的 JAR 复制进去，统一命名为 app.jar
cp target/gd-cors-*.jar docker/app-gd-cors/app.jar
```

在 `docker/app-gd-cors/` 下创建精简版 `Dockerfile`（只有 Run 阶段）：

```bash
cat > docker/app-gd-cors/Dockerfile << 'EOF'
# 直接使用本地编译好的 JAR，不在服务器上重新编译
FROM eclipse-temurin:17-jdk-jammy
WORKDIR /app

COPY app.jar app.jar

ENV LANG=C.UTF-8
ENV LC_ALL=C.UTF-8
ENV JAVA_TOOL_OPTIONS="-Dfile.encoding=UTF-8"

RUN apt-get update && \
    apt-get install -y tzdata && \
    ln -fs /usr/share/zoneinfo/Asia/Shanghai /etc/localtime && \
    echo "Asia/Shanghai" > /etc/timezone && \
    dpkg-reconfigure --frontend noninteractive tzdata && \
    rm -rf /var/lib/apt/lists/*

EXPOSE 8080

ENTRYPOINT ["java", \
    "-server", \
    "-XX:+UnlockExperimentalVMOptions", \
    "-XX:G1NewSizePercent=20", \
    "-XX:InitialRAMPercentage=65.0", \
    "-XX:MaxRAMPercentage=65.0", \
    "-XX:MinRAMPercentage=65.0", \
    "-XX:MaxDirectMemorySize=8g", \
    "-XX:+UseG1GC", \
    "-jar", "app.jar"]
EOF
```

### 4.2 准备前端构建上下文

前端在服务器上通过 Docker 多阶段构建完成（Node 环境在容器内，无需本地安装）：

```bash
cd /path/to/gd-cors

# 将前端源码复制到 docker/ 目录下
cp -r web-gd-cors docker/web-gd-cors
```

### 4.3 完成后的目录结构

```
docker/
├── app-gd-cors/
│   ├── Dockerfile      # 精简版，仅 Run 阶段
│   └── app.jar         # 本地编译好的 JAR
└── web-gd-cors/
    ├── Dockerfile
    ├── package.json
    └── src/
```

### 4.4 将 docker/ 目录上传到服务器

```bash
# 使用 rsync 上传（推荐，支持增量同步）
rsync -avz --progress /path/to/gd-cors/docker/ user@server:/home/gdcors/docker/

# 或使用 scp
scp -r /path/to/gd-cors/docker/ user@server:/home/gdcors/docker/
```

> 后续所有操作均在**服务器**上的 `docker/` 目录下执行。

## 5. 第三步：配置环境变量

所有服务配置均通过 `docker/.env` 文件管理。

```bash
cd docker/

# 编辑 .env 文件（已有默认值，按需修改）
vim .env
```

### 关键配置项说明

#### 存储路径（必须修改）

```dotenv
# 所有持久化数据的根目录，确保该路径所在磁盘有足够空间
STORAGE_PATH_1="/home/gdcors/dk/data-disk/gd-cors"
```

#### 数据库密码

```dotenv
MYSQL_ROOT_PASSWORD=root          # MySQL root 密码
REDIS_ROOT_PASSWORD=redis         # Redis 密码
RABBITMQ_DEFAULT_PASS=rabbitmq    # RabbitMQ 密码
MINIO_ROOT_PASSWORD=minioadmin    # MinIO 密码
```

#### AI 模型配置

```dotenv
# 在线大模型（智谱 GLM）
LANGCHAIN4J_OPENAI_ONLINE_MODEL_ZHIPU_API_KEY=<你的智谱API Key>

# 本地 Ollama 聊天模型（通过 Nginx 负载均衡）
LANGCHAIN4J_OLLAMA_CHAT_BASE_URL=http://host.docker.internal:11434
LANGCHAIN4J_OLLAMA_CHAT_MODEL_NAME=qwen3:32b

# 本地 Ollama 视觉模型
LANGCHAIN4J_OLLAMA_VL_BASE_URL=http://host.docker.internal:11435
LANGCHAIN4J_OLLAMA_VL_MODEL_NAME=qwen3-vl:8b

# 本地 Ollama Embedding 模型
LANGCHAIN4J_OLLAMA_EMBEDDING_BASE_URL=http://host.docker.internal:11436
LANGCHAIN4J_OLLAMA_EMBEDDING_MODEL_NAME=qwen3-embedding:0.6b

# 联网搜索（Tavily，可选）
LANGCHAIN4J_TAVILY_API_KEY=<你的Tavily API Key>
```

#### JWT 安全配置

```dotenv
JWT_SECURITY=cors                 # JWT 签名密钥，生产环境请修改为复杂字符串
JWT_ACCESS_EXPIRE=7200000         # Access Token 有效期（毫秒），默认 2 小时
JWT_REFRESH_EXPIRE=2592000000     # Refresh Token 有效期（毫秒），默认 30 天
```

## 6. 第四步：初始化存储目录与权限

运行 `full_auth.sh` 脚本，自动创建所有必要目录并设置正确的容器用户权限。

```bash
cd docker/

# 赋予执行权限
chmod +x full_auth.sh

# 执行（需要 sudo）
sudo bash full_auth.sh
```

脚本会完成以下操作：

| 目录 | 容器用户 UID | 权限 |
|------|------------|------|
| `$STORAGE_PATH_1/mysql` | 999 | 750 |
| `$STORAGE_PATH_1/redis` | 999 | 750 |
| `$STORAGE_PATH_1/rabbitmq` | 999 | 750 |
| `$STORAGE_PATH_1/minio` | 1000 | 750 |
| `$STORAGE_PATH_1/milvus` | 1000 | 750 |
| `$STORAGE_PATH_1/tei` | 1000 | 755 |
| `$STORAGE_PATH_1/app/logs` | 1000 | 755 |

> **注意**：每次重新部署前建议重新执行此脚本，确保权限正确。

## 7. 第五步：启动 Ollama 模型服务

Ollama 运行在宿主机（非容器内），通过 `host.docker.internal` 供容器访问。

### 7.1 安装 Ollama

```bash
# Linux 安装
curl -fsSL https://ollama.com/install.sh | sh
```

### 7.2 拉取所需模型

```bash
# 聊天推理模型
ollama pull qwen3:32b

# 视觉理解模型
ollama pull qwen3-vl:8b

# 向量化模型
ollama pull qwen3-embedding:0.6b
```

### 7.3 启动多 GPU 实例

```bash
cd docker/

chmod +x start_ollama.sh
bash start_ollama.sh
```

脚本会在以下端口启动 Ollama 实例：

| 端口 | GPU | 模型 | 用途 |
|------|-----|------|------|
| 11431 | GPU 0,1 | qwen3:32b | 聊天推理 |
| 11432 | GPU 2,3 | qwen3:32b | 聊天推理 |
| 11433 | GPU 4,5 | qwen3:32b | 聊天推理 |
| 11435 | GPU 6 | qwen3-vl:8b | 视觉理解 |
| 11436 | GPU 7 | qwen3-embedding:0.6b | 向量化 |

> **单 GPU / CPU 环境**：修改 `start_ollama.sh`，只启动一个实例，并相应修改 `.env` 中的 `LANGCHAIN4J_OLLAMA_*_BASE_URL`。

### 7.4 验证 Ollama 服务

```bash
# 验证聊天实例（通过 Nginx 负载均衡）
curl http://localhost:11434/api/tags

# 验证 Embedding 实例
curl http://localhost:11436/api/tags
```

## 8. 第六步：启动所有 Docker 服务

```bash
cd docker/

# 首次启动（会构建镜像，耗时较长）
docker compose up -d --build

# 非首次启动（镜像已存在）
docker compose up -d
```

### 启动顺序说明

Docker Compose 会按依赖关系自动排序启动：

```
基础设施层（并行启动）
  ├── gd-cors-mysql
  ├── gd-cors-redis
  ├── gd-cors-rabbitmq
  ├── gd-cors-minio
  └── gd-cors-milvus-etcd
        └── gd-cors-milvus（依赖 etcd + minio）

AI 服务层
  ├── gd-cors-nginx-lb（Ollama 负载均衡）
  └── gd-cors-tei-reranker（重排序服务，需 GPU）

应用层（依赖所有基础设施）
  ├── gd-cors-app-1（Spring Boot 实例 1）
  └── gd-cors-app-2（Spring Boot 实例 2）

前端层
  └── gd-cors-web（Vue3 + Nginx）
```

## 10. 增量更新流程

> 适用于代码变更后的滚动更新，无需重建所有服务。

### 10.1 仅更新后端

```bash
# --- 在本地开发机上执行 ---

# 1. 重新编译 JAR
cd /path/to/gd-cors
mvn clean package -DskipTests

# 2. 替换 docker/app-gd-cors/ 下的 JAR
cp target/gd-cors-*.jar docker/app-gd-cors/app.jar

# 3. 将新 JAR 上传到服务器（增量同步，只传变更文件）
rsync -avz docker/app-gd-cors/app.jar user@server:/home/gdcors/docker/app-gd-cors/app.jar

# --- 在服务器上执行 ---

cd /home/gdcors/docker/

# 4. 重新构建镜像并重启后端实例（不影响其他服务）
docker compose up -d --build gd-cors-app-1 gd-cors-app-2

# 5. 观察启动日志
docker compose logs -f gd-cors-app-1 gd-cors-app-2
```

### 10.2 仅更新前端

```bash
# --- 在本地开发机上执行 ---

# 1. 将最新前端源码上传到服务器（排除 node_modules 和 dist）
rsync -avz --exclude='node_modules' --exclude='dist' \
  /path/to/gd-cors/web-gd-cors/ \
  user@server:/home/gdcors/docker/web-gd-cors/

# --- 在服务器上执行 ---

cd /home/gdcors/docker/

# 2. 重新构建并重启前端
docker compose up -d --build gd-cors-web

# 3. 验证
curl -I http://localhost:80
```

### 10.3 更新数据库 Schema（增量 SQL）

> 当 `schema.sql` 有新增表或字段时，需手动执行增量 SQL。

```bash
# 方式一：通过 docker exec 执行 SQL 文件
docker compose exec -T gd-cors-mysql \
  mysql -u root -proot gd_cors < ./mysql/init/schema.sql

# 方式二：进入 MySQL 容器手动执行
docker compose exec gd-cors-mysql mysql -u root -proot gd_cors
```

> **注意**：`schema.sql` 中所有建表语句均使用 `CREATE TABLE IF NOT EXISTS`，重复执行安全，不会覆盖已有数据。

> **关于数据安全**：MySQL 官方镜像只在数据目录**为空**时才执行初始化脚本（`/docker-entrypoint-initdb.d/`）。若 `$STORAGE_PATH_1/mysql/` 已有数据，重新 `docker compose up` 不会触发初始化，原有数据完全保留。

### 10.4 更新环境变量配置

```bash
cd docker/

# 1. 修改 .env 文件
vim .env

# 2. 重启受影响的服务（以后端为例）
docker compose up -d gd-cors-app-1 gd-cors-app-2

# 若修改了基础设施配置（如 Redis 密码），需重启对应服务
docker compose up -d gd-cors-redis
```

### 10.5 完整重建（慎用）

```bash
cd docker/

# 停止并删除所有容器（数据卷不受影响，宿主机 $STORAGE_PATH_1/ 下的数据完全保留）
docker compose down

# 重新构建所有镜像并启动
docker compose up -d --build
```

> **数据安全说明**：`docker compose down` 不加 `-v` / `--volumes` 参数，**不会删除任何持久化数据**。所有数据通过 Bind Mount 挂载到宿主机 `$STORAGE_PATH_1/` 目录，容器删除后数据仍在磁盘上，重新启动后自动恢复。如需彻底清空数据，需手动删除 `$STORAGE_PATH_1/` 下对应子目录（**高危操作，请谨慎**）。

## 13. 常见问题排查

### Q1：MySQL 容器启动失败，提示权限错误

```bash
# 修复 MySQL 数据目录权限
sudo chown -R 999:999 ${STORAGE_PATH_1}/mysql
sudo chmod -R 750 ${STORAGE_PATH_1}/mysql

# 重启 MySQL
docker compose restart gd-cors-mysql
```

### Q2：Redis 启动失败，AOF 目录不存在

```bash
sudo mkdir -p ${STORAGE_PATH_1}/redis/appendonlydir
sudo chown -R 999:999 ${STORAGE_PATH_1}/redis
sudo chmod 770 ${STORAGE_PATH_1}/redis/appendonlydir

docker compose restart gd-cors-redis
```

### Q3：RabbitMQ 启动失败，Cookie 权限问题

```bash
# 删除旧 Cookie，让 RabbitMQ 重新生成
sudo rm -f ${STORAGE_PATH_1}/rabbitmq/.erlang.cookie
sudo chown -R 999:999 ${STORAGE_PATH_1}/rabbitmq
sudo chmod -R 750 ${STORAGE_PATH_1}/rabbitmq

docker compose restart gd-cors-rabbitmq
```

### Q4：后端应用无法连接 Ollama

检查 Ollama 是否在宿主机正常运行：

```bash
# 检查 Ollama 进程
ps aux | grep ollama

# 检查端口监听
ss -tlnp | grep 1143

# 容器内测试连通性
docker compose exec gd-cors-app-1 curl http://host.docker.internal:11434/api/tags
```

### Q5：Milvus 启动失败

```bash
# 检查 etcd 是否健康
docker compose exec gd-cors-milvus-etcd etcdctl endpoint health

# 检查 MinIO 是否健康
curl http://localhost:9000/minio/health/live

# 查看 Milvus 日志
docker compose logs gd-cors-milvus
```

### Q6：前端页面空白或 API 请求失败

检查前端 Nginx 配置中的后端代理地址是否正确，以及后端服务是否正常运行：

### Q7：本地 Maven 编译时依赖下载超时

在本地 `~/.m2/settings.xml` 中配置阿里云镜像加速：

```xml
<settings>
  <mirrors>
    <mirror>
      <id>aliyun</id>
      <mirrorOf>central</mirrorOf>
      <url>https://maven.aliyun.com/repository/public</url>
    </mirror>
  </mirrors>
</settings>
```
