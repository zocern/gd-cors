## 1. 功能概述

在原有文件管理基础上，追加**标签管理**能力：

- **标签 CRUD**：支持创建、更新、删除、查询标签，标签名称在**同项目内唯一**（不同项目可同名），可设置颜色和描述。
  - 查询支持两种模式：非分页列表（`GET /tags`，最多 1000 条）和分页（`GET /tags/page`）。
  - 查询时可按 `project` 过滤（只看某项目的标签），也可不传查全表。
- **文件绑定标签**：上传文件时可同时绑定标签；也可在上传后追加绑定、全量更新或解绑标签。
- **文件夹绑定标签**：标签不仅限于文件，文件夹同样支持绑定标签。
- **标签维度查询**：
  - 按标签分页查询绑定的文件/文件夹列表（`GET /tags/{id}/files`）。
  - 查询文件列表时支持按标签过滤（`GET /files?tag-id={tagId}`，最多 1000 条）。
  - 查询文件详情时可附带标签信息（`GET /files/{id}?with-tags=true`）。
  - 查询单个文件/文件夹绑定的所有标签（`GET /files/{id}/tags`）。
- **并发安全**：
  - 文件标签的绑定/更新/解绑使用**文件维度分布式锁**（`lock:file-tag:file:{fileId}`）保证串行化。
  - 标签删除使用**标签维度分布式锁**（`lock:tag:{tagId}`）避免与并发绑定操作产生孤儿记录。
- **数据一致性**：删除文件/文件夹时**级联清理** `file_tag_relation` 表中的相关记录，避免孤儿。

## 2. 数据库变更

### 2.1 新增 `tags` 标签表

```sql
CREATE TABLE IF NOT EXISTS `tags`
(
    `id`          BIGINT UNSIGNED AUTO_INCREMENT COMMENT '标签ID，主键' PRIMARY KEY,
    `project`     VARCHAR(100)                        NOT NULL COMMENT '所属项目标识（同项目内标签名唯一）',
    `name`        VARCHAR(100)                        NOT NULL COMMENT '标签名称（同项目内唯一）',
    `color`       VARCHAR(20)                         NULL COMMENT '标签颜色（如 #FF5733）',
    `description` VARCHAR(500)                        NULL COMMENT '标签描述',
    `created_by`  BIGINT                              NULL COMMENT '创建者ID',
    `updated_by`  BIGINT                              NULL COMMENT '更新者ID',
    `created`     DATETIME DEFAULT CURRENT_TIMESTAMP  NOT NULL COMMENT '创建时间',
    `updated`     DATETIME DEFAULT CURRENT_TIMESTAMP  NOT NULL ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    CONSTRAINT `uk_tag_project_name` UNIQUE (`project`, `name`),
    INDEX `idx_tag_project` (`project`)
)
    COMMENT '标签表'
    ENGINE = InnoDB
    DEFAULT CHARSET = utf8mb4
    COLLATE = utf8mb4_unicode_ci;
```

> **设计说明**：
> - `project` 字段为项目标识字符串（必填），用于将标签按项目隔离。
> - 唯一约束从 `name` 全局唯一改为 `(project, name)` 联合唯一，不同项目允许同名标签。
> - `idx_tag_project` 索引加速按项目过滤查询。

### 2.2 新增 `file_tag_relation` 文件标签关联表

```sql
CREATE TABLE IF NOT EXISTS `file_tag_relation`
(
    `id`       BIGINT UNSIGNED AUTO_INCREMENT COMMENT '主键ID' PRIMARY KEY,
    `file_id`  BIGINT UNSIGNED NOT NULL COMMENT '关联的 file_metadata.id（文件或文件夹）',
    `tag_id`   BIGINT UNSIGNED NOT NULL COMMENT '关联的 tags.id',
    `created`  DATETIME DEFAULT CURRENT_TIMESTAMP NOT NULL COMMENT '绑定时间',
    UNIQUE KEY `uk_file_tag` (`file_id`, `tag_id`),
    INDEX `idx_file_id`  (`file_id`),
    INDEX `idx_tag_id`   (`tag_id`)
)
    COMMENT '文件标签关联表'
    ENGINE = InnoDB
    DEFAULT CHARSET = utf8mb4
    COLLATE = utf8mb4_unicode_ci;
```

> **设计说明**：
> - `file_id` 对应 `file_metadata.id`，文件和文件夹均可绑定标签。
> - `uk_file_tag` 唯一索引保证同一文件不会重复绑定同一标签。
> - `INSERT IGNORE` 策略保证并发绑定的幂等性。

### 2.3 `file_metadata` 表无需变更

标签信息通过关联表维护，`file_metadata` 表结构不变。

## 3. 新增 / 修改接口

### 3.1 标签管理接口

所有接口前缀：`/api/v1/tags`，需登录（携带 `Authorization` Header）。

#### `POST /tags` — 创建标签

**请求体**

```json
{
  "project": "project-alpha",
  "name": "重要文件",
  "color": "#FF5733",
  "description": "标记重要文件"
}
```

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| project | String | 是 | 项目标识（≤100字符，同项目内标签名唯一） |
| name | String | 是 | 标签名称（≤100字符，同项目内唯一） |
| color | String | 否 | 标签颜色（如 `#FF5733`，≤20字符） |
| description | String | 否 | 标签描述（≤500字符） |

**响应**：HTTP 201 Created

```json
{
  "code": 200,
  "data": 1001
}
```

> `data` 为新创建的标签ID。

**响应示例**

**新增请求参数**

| 参数 | 位置 | 类型 | 必填 | 说明 |
|------|------|------|------|------|
| tag-id | Query | Long | 否 | 按标签ID过滤，返回该标签下的文件/文件夹（分页） |
| pageNum | Query | Integer | 否 | 页码，默认 1（仅 `tag-id` 存在时生效） |
| pageSize | Query | Integer | 否 | 每页大小，默认 10（仅 `tag-id` 存在时生效） |

## 4. 核心流程说明

### 4.1 创建标签

```
POST /tags  { "name": "重要文件", "color": "#FF5733" }
    │
    ▼
校验标签名称唯一性
    │
    ▼
插入 tags 表
    │
    ▼
返回新标签ID
```

### 4.2 上传文件时绑定标签

```
POST /files/upload?tag-ids=1001&tag-ids=1002
    │
    ▼
【Controller 预校验】tagService.validateTagsExist(tagIds)
    └── 若标签不存在则立即抛 NotFoundException（文件未上传，无残留）
    │
    ▼
fileMetadataService.uploadFile(file, parentId)
    ├── 上传文件到 MinIO
    └── saveFileRecordWithTransaction()  [事务]
        ├── 插入 file_metadata
        ├── 插入 file_versions（v1）
        └── 插入 file_vector_status（PENDING）
    │   事务提交后立即发送 FileVectorMessage → 异步向量化
    │   （若 MinIO 上传成功但 DB 失败，发送 MinIO 回滚消息）
    │
    ▼
tagService.bindTags(fileId, tagIds)  【独立事务 + 文件维度分布式锁】
    └── INSERT IGNORE INTO file_tag_relation（幂等）
```

> **设计说明**：
> - 上传前的"预校验"和上传后的"标签绑定"是两个独立步骤，互不在同一事务内。
> - 通过先校验保证标签合法，避免文件已落库但标签绑定失败的尴尬场景。
> - `bindTags` 内部（`bindTagsInTransaction`）也会再次校验标签是否存在，形成双重保障，防止标签在预校验通过后、绑定前被并发删除。
> - 标签绑定通过 `lock:file-tag:file:{fileId}` 分布式锁保证同一文件的并发安全。

### 4.3 追加绑定标签

```
POST /files/{id}/tags  { "tagIds": [1001, 1002] }
    │
    ▼
获取文件维度分布式锁 lock:file-tag:file:{fileId}
    │
    ▼
[事务开始]
    ├── 校验文件/文件夹存在
    ├── 校验标签全部存在（Service 内部再次校验，双重保障）
    └── INSERT IGNORE INTO file_tag_relation（幂等，重复绑定不报错）
[事务提交]
    │
    ▼
释放锁
```

### 4.4 全量更新标签绑定

```
PUT /files/{id}/tags  { "tagIds": [1003] }
    │
    ▼
获取文件维度分布式锁 lock:file-tag:file:{fileId}
    │
    ▼
[事务开始]
    ├── 校验文件/文件夹存在
    ├── 校验标签全部存在
    ├── DELETE FROM file_tag_relation WHERE file_id = ?  [清空旧绑定]
    └── INSERT IGNORE INTO file_tag_relation（重新绑定）
[事务提交]
    │
    ▼
释放锁
```

### 4.5 删除标签（级联清理 + 标签维度加锁）

```
DELETE /tags/{id}
    │
    ▼
获取标签维度分布式锁 lock:tag:{tagId}
（避免删除时其他请求正在 bindTags 该标签，导致孤儿 relation）
    │
    ▼
[事务开始]
    ├── DELETE FROM file_tag_relation WHERE tag_id = ?  [解除所有文件绑定]
    └── DELETE FROM tags WHERE id = ?
[事务提交]
    │
    ▼
释放锁
```

### 4.5.1 删除文件时级联清理标签关联（重要）

```
DELETE /files/{id}
    │
    ▼
[deleteWithTransaction]
    ├── 收集所有要删除的 fileId（文件夹时递归收集子孙）
    ├── 收集所有版本 storageKey
    ├── DELETE FROM file_versions WHERE file_id IN (...)
    ├── DELETE FROM file_tag_relation WHERE file_id IN (...)  【新增】
    ├── DELETE FROM file_metadata WHERE id IN (...)
    └── DELETE FROM file_association WHERE ...
    │
    ▼
事务提交后：
    └── 清理 MinIO 文件 + Milvus 向量
```

### 4.6 按标签查询文件

```
GET /tags/{id}/files?pageNum=1&pageSize=10
    │
    ▼
JOIN file_metadata + file_tag_relation
    │
    ▼
分页返回文件/文件夹列表
```

## 6. 存量数据无缝迁移方案

### 6.1 整体原则

- **不停机**：所有变更均为新增表，不修改现有表结构。
- **不丢数据**：存量文件/文件夹无标签绑定，功能完全向后兼容。
- **分步执行**：先执行 DDL，再部署新代码。

### 6.2 执行步骤

#### Step 1：执行 DDL（可在线执行，不锁表）

```sql
-- 1. 创建标签表
CREATE TABLE IF NOT EXISTS `tags`
(
    `id`          BIGINT UNSIGNED AUTO_INCREMENT COMMENT '标签ID，主键' PRIMARY KEY,
    `project`     VARCHAR(100)                        NOT NULL COMMENT '所属项目标识（同项目内标签名唯一）',
    `name`        VARCHAR(100)                        NOT NULL COMMENT '标签名称（同项目内唯一）',
    `color`       VARCHAR(20)                         NULL COMMENT '标签颜色（如 #FF5733）',
    `description` VARCHAR(500)                        NULL COMMENT '标签描述',
    `created_by`  BIGINT                              NULL COMMENT '创建者ID',
    `updated_by`  BIGINT                              NULL COMMENT '更新者ID',
    `created`     DATETIME DEFAULT CURRENT_TIMESTAMP  NOT NULL COMMENT '创建时间',
    `updated`     DATETIME DEFAULT CURRENT_TIMESTAMP  NOT NULL ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    CONSTRAINT `uk_tag_project_name` UNIQUE (`project`, `name`),
    INDEX `idx_tag_project` (`project`)
) ENGINE = InnoDB DEFAULT CHARSET = utf8mb4 COLLATE = utf8mb4_unicode_ci COMMENT '标签表';

-- 2. 创建文件标签关联表
CREATE TABLE IF NOT EXISTS `file_tag_relation`
(
    `id`       BIGINT UNSIGNED AUTO_INCREMENT COMMENT '主键ID' PRIMARY KEY,
    `file_id`  BIGINT UNSIGNED NOT NULL COMMENT '关联的 file_metadata.id（文件或文件夹）',
    `tag_id`   BIGINT UNSIGNED NOT NULL COMMENT '关联的 tags.id',
    `created`  DATETIME DEFAULT CURRENT_TIMESTAMP NOT NULL COMMENT '绑定时间',
    UNIQUE KEY `uk_file_tag` (`file_id`, `tag_id`),
    INDEX `idx_file_id`  (`file_id`),
    INDEX `idx_tag_id`   (`tag_id`)
) ENGINE = InnoDB DEFAULT CHARSET = utf8mb4 COLLATE = utf8mb4_unicode_ci COMMENT '文件标签关联表';
```

## 附：标签与文件关系示意

```
标签: "重要文件" (#FF5733)
  ├── 绑定文件: 项目报告.pdf (id=100)
  ├── 绑定文件: 合同文件.docx (id=101)
  └── 绑定文件夹: 2026年项目/ (id=50)

标签: "待审核" (#FFA500)
  ├── 绑定文件: 设计方案.pdf (id=102)
  └── 绑定文件: 项目报告.pdf (id=100)  ← 同一文件可绑定多个标签

文件: 项目报告.pdf (id=100)
  ├── 标签: "重要文件"
  └── 标签: "待审核"
```

> **设计原则**：标签与文件/文件夹是多对多关系，一个文件可绑定多个标签，一个标签可绑定多个文件/文件夹。
